Week 4 (JavaScript): JavaScript Fundamentals & Secure Coding

📅 Duration: 2 Days (16 Hours) 🔬 Labs: 13 Hours 📖 Theory: 3 Hours 🎯 Level 1 – Week 4
Module Overview: This week introduces JavaScript — the programming language that makes web pages interactive. Students move from static HTML and CSS into live, responsive behaviour: responding to button clicks, reading input, updating the page without reloading, and validating data before it is submitted. The second day connects every concept to security: understanding why client-side code can never be trusted alone, how Cross-Site Scripting (XSS) attacks work, and which JavaScript patterns to avoid entirely in professional code. All theory is kept short and immediately followed by hands-on practice, since 13 of the 16 hours this week are lab time.
🎯 Week Objectives — By the End of This Week Students Will Be Able To:
  1. Declare variables with let and const, work with the five primitive data types, and write conditional logic and loops
  2. Write named functions, arrow functions, and understand scope and return values
  3. Select, read, modify, and create HTML elements using the Document Object Model (DOM)
  4. Attach event listeners to respond to user interactions (clicks, keypresses, form submissions)
  5. Build a fully functional interactive calculator from scratch using only HTML, CSS, and JavaScript
  6. Implement real-time client-side form validation with clear, accessible error messaging
  7. Build a complete two-player Tic-Tac-Toe game with win detection and reset functionality
  8. Explain why client-side validation must always be paired with server-side validation
  9. Identify and avoid dangerous JavaScript patterns: eval(), innerHTML with user input, and document.write()
  10. Prevent DOM-based XSS by using safe DOM methods and sanitising user-supplied data before rendering

Day 1 — JavaScript Fundamentals & DOM Interaction 8 Hours

Theory Session 2 Hours

Theory is taught in four back-to-back segments with live coding throughout. Students follow along in a plain .html file open in their browser. Every concept shown in theory is used in the labs that follow the same day.

1. Variables, Data Types & Operators 30 min

JavaScript runs inside the browser and can read and change anything on the page in real time. Before that, students need the building blocks: how to store data, what types of data exist, and how to do arithmetic and comparisons. The instructor types each example into the browser console so students see the output instantly.

// --- VARIABLES --- // const: value can never be reassigned (prefer this by default) const studentName = "Amara"; // let: value can change over time let score = 0; // --- THE FIVE PRIMITIVE TYPES --- const age = 22; // number const name = "Fatima"; // string const isActive = true; // boolean const nothing = null; // null — intentionally empty let missing; // undefined — declared but not yet set // typeof tells you what type a value is — useful for validation later console.log(typeof age); // "number" console.log(typeof name); // "string" console.log(typeof isActive); // "boolean" // --- ARITHMETIC --- console.log(10 + 3); // 13 console.log(10 - 3); // 7 console.log(10 * 3); // 30 console.log(10 / 3); // 3.3333... console.log(10 % 3); // 1 (remainder / modulus) // --- COMPARISONS (always use === not ==) --- console.log(5 === 5); // true console.log(5 === "5"); // false — number vs string, strict equality console.log(5 == "5"); // true — DO NOT USE ==, it coerces types unsafely // --- CONDITIONALS --- if (score >= 50) { console.log("Pass"); } else if (score >= 40) { console.log("Borderline"); } else { console.log("Fail"); } // --- LOOPS --- for (let i = 1; i <= 5; i++) { console.log("Count: " + i); // Count: 1 Count: 2 ... Count: 5 } // Arrays — ordered lists of values const students = ["Amara", "Emeka", "Fatima"]; console.log(students[0]); // "Amara" (index starts at 0) console.log(students.length); // 3 students.forEach(s => console.log(s)); // Amara Emeka Fatima
Security Note — Always Use ===: JavaScript's loose equality operator == silently converts types before comparing. This has caused real security bugs where an attacker sends a string that loose-equals a number used in an access check. As a rule, always use strict equality === throughout this course. This is the first of several secure-by-default habits.

2. Functions 30 min

Functions group a set of instructions under one name so they can be reused. Every interactive feature in the labs is built around functions — understanding how to write them, call them, pass data in, and get data out is the most important skill of Day 1.

// --- NAMED FUNCTION (traditional style) --- function greet(name) { return "Hello, " + name + "!"; } console.log(greet("Amara")); // "Hello, Amara!" // --- ARROW FUNCTION (modern, compact style) --- const add = (a, b) => a + b; console.log(add(4, 6)); // 10 // --- FUNCTION WITH MULTIPLE STEPS --- function calculateGrade(score) { if (score >= 70) return "Distinction"; if (score >= 50) return "Pass"; return "Fail"; } console.log(calculateGrade(85)); // "Distinction" console.log(calculateGrade(55)); // "Pass" console.log(calculateGrade(30)); // "Fail" // --- SCOPE: variables declared inside a function stay inside --- function example() { const secret = "only visible in here"; console.log(secret); // works fine } example(); console.log(secret); // ReferenceError: secret is not defined

3. Document Object Model (DOM) Manipulation 30 min

The DOM is the browser's live representation of the HTML on the page. JavaScript can read, modify, create, and delete any element at any time. This is what makes a page interactive — instead of reloading, the page updates in place. The instructor demonstrates each method live, with students watching the page change in real time.

// ---- SELECTING ELEMENTS ---- // Select one element by its id attribute const title = document.getElementById("main-title"); // Select the FIRST element matching a CSS selector const btn = document.querySelector("button"); // Select ALL elements matching a CSS selector (returns a NodeList) const items = document.querySelectorAll(".menu-item"); // ---- READING AND CHANGING CONTENT ---- console.log(title.textContent); // read the text inside the element title.textContent = "New Title"; // change the text safely // ---- CHANGING STYLES ---- title.style.color = "#E8931F"; title.style.fontSize = "2rem"; // ---- TOGGLING CSS CLASSES (preferred over inline styles) ---- btn.classList.add("active"); btn.classList.remove("active"); btn.classList.toggle("hidden"); // adds if absent, removes if present // ---- READING INPUT VALUES ---- const emailInput = document.querySelector("#email"); const userEmail = emailInput.value; // whatever the user has typed // ---- CREATING AND ADDING ELEMENTS ---- const newPara = document.createElement("p"); newPara.textContent = "This paragraph was created by JavaScript."; document.body.appendChild(newPara); // adds it to the end of <body>

4. Event Listeners 30 min

An event listener tells the browser: "when this happens, run this function." Almost all interactivity comes down to attaching the right listener to the right element. The instructor builds a live counter on screen — students see the number increment with every click.

// ---- BASIC CLICK LISTENER ---- const button = document.querySelector("#my-btn"); button.addEventListener("click", function() { console.log("Button was clicked!"); }); // ---- LIVE COUNTER — combining DOM + events + state variable ---- let count = 0; const display = document.querySelector("#count-display"); const countBtn = document.querySelector("#count-btn"); countBtn.addEventListener("click", () => { count++; display.textContent = count; // page updates instantly — no reload }); // ---- FORM SUBMISSION EVENT ---- const form = document.querySelector("#signup-form"); form.addEventListener("submit", (event) => { event.preventDefault(); // STOP the page from reloading on submit const email = document.querySelector("#email").value; console.log("Submitted email: " + email); }); // ---- INPUT EVENT — fires on every keystroke ---- const nameField = document.querySelector("#name"); nameField.addEventListener("input", () => { console.log("Current value: " + nameField.value); }); // ---- COMMON EVENT TYPES ---- // "click" — mouse click // "submit" — form submitted // "input" — input field changes (every keystroke) // "change" — input loses focus after changing // "keydown" — any key is pressed // "mouseover" — mouse moves over an element // "DOMContentLoaded" — page HTML has fully loaded
Key Takeaway from Theory: Every interactive feature follows the same pattern — Select an element, Listen for an event, Read or update data, Update the DOM. Students who understand this four-step loop can build almost anything. The labs today use this loop repeatedly until it becomes instinctive.

Lab 1 — Interactive Calculator 3 Hours

🎯 Lab 1 Objective

Build a fully working calculator from scratch that performs addition, subtraction, multiplication, and division. The calculator must: display a running input as the user clicks digit and operator buttons; compute and display the correct result on pressing equals; handle division by zero gracefully; and include a clear button that resets all state. No starter code is provided — students write everything.

Part A — HTML Structure (20 min)

Start with the HTML layout. The calculator needs a display area and a grid of buttons. Write the HTML first with no JavaScript — load it in the browser and confirm it renders before adding any logic.

<!-- calculator.html — build this structure first, style second --> <div id="calculator"> <div id="display">0</div> <div id="buttons"> <!-- Row 1: Clear and operators --> <button class="btn-clear" id="clear">C</button> <button class="btn-operator" data-op="/">÷</button> <button class="btn-operator" data-op="*">×</button> <button class="btn-operator" data-op="-">−</button> <!-- Rows 2-4: Digit buttons (use data-digit attribute) --> <button class="btn-digit" data-digit="7">7</button> <button class="btn-digit" data-digit="8">8</button> <button class="btn-digit" data-digit="9">9</button> <!-- ... continue for 4,5,6, then 1,2,3, then 0 and decimal --> <button class="btn-operator" data-op="+">+</button> <button id="equals">=</button> </div> </div> <script src="calculator.js"></script>
Part B — State Variables & Display Logic (30 min)

Create calculator.js. The calculator needs to remember three things between button clicks: the first number entered, the operator chosen, and the current display value. These are the state variables. Write the state and the display update function before attaching any events.

// calculator.js // State — these three variables hold everything the calculator remembers let firstOperand = ""; // the number entered before the operator let operator = ""; // the chosen operator: +, -, *, / let currentInput = "0"; // what is currently shown on the display let justCalculated = false; // true after = is pressed, so next digit starts fresh const display = document.getElementById("display"); // Always update the display through this function — never write to it directly function updateDisplay(value) { display.textContent = value; }
Part C — Digit, Operator & Equals Logic (60 min)

Write three functions — one for digit presses, one for operator presses, one for equals — then attach event listeners to every button. Use data-digit and data-op attributes to keep the JavaScript clean and avoid 17 separate click handlers.

// --- DIGIT BUTTON PRESSED --- function handleDigit(digit) { if (justCalculated) { currentInput = digit; // start fresh after = was pressed justCalculated = false; } else { currentInput = (currentInput === "0") ? digit : currentInput + digit; } updateDisplay(currentInput); } // --- OPERATOR BUTTON PRESSED --- function handleOperator(op) { firstOperand = currentInput; // save what was entered before the operator operator = op; currentInput = "0"; // ready for the second number } // --- EQUALS PRESSED — compute the result --- function handleEquals() { const a = parseFloat(firstOperand); const b = parseFloat(currentInput); let result; switch (operator) { case "+": result = a + b; break; case "-": result = a - b; break; case "*": result = a * b; break; case "/": if (b === 0) { updateDisplay("Error: ÷0"); // graceful error — never crash handleClear(); return; } result = a / b; break; default: return; } updateDisplay(result); currentInput = String(result); justCalculated = true; } // --- CLEAR --- function handleClear() { firstOperand = ""; operator = ""; currentInput = "0"; justCalculated = false; updateDisplay("0"); } // --- ATTACH EVENT LISTENERS --- // All digit buttons in one loop — reads the data-digit attribute document.querySelectorAll(".btn-digit").forEach(btn => { btn.addEventListener("click", () => handleDigit(btn.dataset.digit)); }); // All operator buttons in one loop — reads the data-op attribute document.querySelectorAll(".btn-operator").forEach(btn => { btn.addEventListener("click", () => handleOperator(btn.dataset.op)); }); document.getElementById("equals").addEventListener("click", handleEquals); document.getElementById("clear").addEventListener("click", handleClear);
Part D — Decimal Support & Edge Cases (30 min)

Extend the calculator to handle decimal numbers, and test every edge case before finishing. The instructor walks through each test scenario — students run them in their own browser and fix anything that does not behave correctly.

  • Enter 5.5 + 2.2 = — should display 7.7
  • Press decimal twice on the same number — should not add a second decimal point
  • Press 8 ÷ 0 = — should show Error: ÷0 and reset cleanly
  • Press = = = repeatedly — should not crash or produce unexpected results
  • Press an operator with no number entered — should handle gracefully
  • Enter a very long number — display should not overflow its container
// Add to handleDigit to prevent a second decimal point on the same number function handleDecimal() { if (currentInput.includes(".")) return; // already has one — do nothing currentInput += "."; updateDisplay(currentInput); } document.getElementById("decimal").addEventListener("click", handleDecimal);
Part E — Styling & Peer Test (40 min)

Add CSS to make the calculator visually complete — a dark background, a clear display area, buttons that respond visually to hover and active states. Then swap browsers with the person next to you and try to break each other's calculator. Any bug found is fixed by the original author before the lab ends.

Day 2 — JavaScript Security & Secure Coding Patterns 8 Hours

Theory Session 1 Hour

1. Client-Side Validation — What It Can and Cannot Do 20 min

The form validation built in Lab 2 is a good user experience feature — it gives instant feedback without a page reload. But it provides zero security on its own. Any developer tool, browser extension, or automated script can submit data to a server while bypassing the JavaScript on the page entirely. The only trustworthy validation is the validation that runs on the server, where the attacker has no access.

Client-side validation answers the question: "Is this input sensible?" Server-side validation answers the question: "Is this input safe?" Both are always required. Neither replaces the other. This is the most important concept of Day 2.

Demonstration: The instructor opens the browser DevTools console and submits a value directly to the form's submit event handler by calling it with manually constructed data — completely bypassing every input and blur listener. Students see that the entire client-side validation is invisible from the server's perspective. This is why server-side validation exists.

2. Cross-Site Scripting (XSS) — What It Is and How JavaScript Enables It 25 min

Cross-Site Scripting (XSS) is an attack where malicious JavaScript is injected into a web page and executed in another user's browser. The attacker's script runs with the same trust level as the legitimate website — it can read cookies, steal session tokens, redirect users, or silently capture keystrokes.

The most common way XSS enters a page through JavaScript is via innerHTML. When a developer takes a string from user input — a search term, a comment, a name — and assigns it directly to innerHTML, the browser parses it as HTML. If that string contains a <script> tag or an event handler attribute, the browser executes it.

// DANGEROUS — innerHTML with user input causes XSS const userInput = document.getElementById("search").value; // If the user types: <img src=x onerror="alert('XSS attack!')"> // The browser executes the JavaScript inside onerror document.getElementById("results").innerHTML = "You searched for: " + userInput; // ↑ NEVER do this with user input ↑ DANGEROUS
// SAFE — textContent never interprets HTML, it displays it as plain text document.getElementById("results").textContent = "You searched for: " + userInput; // Even if the user typed <script>...</script>, it shows as literal text

3. Dangerous JavaScript Patterns to Avoid 15 min

Three JavaScript features are responsible for a disproportionate number of security vulnerabilities. Students must understand what each one does and have a safe alternative ready:

// ══════════════════════════════════════════════════════ // 1. eval() — executes a string as JavaScript code // ══════════════════════════════════════════════════════ // DANGEROUS — if userInput contains malicious code, eval() runs it const userInput = "alert('Your session has been stolen')"; eval(userInput); // ← executes the attacker's code directly // SAFE ALTERNATIVE — never use eval() for user input // For maths: use a proper expression parser, not eval() // The calculator in Lab 1 avoided eval() by design — this is why // ══════════════════════════════════════════════════════ // 2. innerHTML with user data — executes embedded HTML/JS // ══════════════════════════════════════════════════════ // DANGEROUS element.innerHTML = userInput; // SAFE ALTERNATIVES element.textContent = userInput; // plain text only — always safe element.setAttribute("title", userInput); // safe for attributes // If you genuinely need HTML from user content (e.g. a rich-text editor): // sanitise it first with a trusted library like DOMPurify element.innerHTML = DOMPurify.sanitize(userInput); // library strips dangerous tags // ══════════════════════════════════════════════════════ // 3. document.write() — overwrites the whole page // ══════════════════════════════════════════════════════ // DANGEROUS — also interpreted as HTML, also executes scripts document.write("<h1>" + userInput + "</h1>"); // SAFE ALTERNATIVE — use DOM methods to create and add elements const heading = document.createElement("h1"); heading.textContent = userInput; // safe document.body.appendChild(heading);
The Golden Rule: Any data that came from a user, a URL parameter, a database, or any external source is untrusted. Before rendering it on the page, always use textContent instead of innerHTML, never pass it to eval(), and never pass it to document.write(). These three rules alone prevent the majority of DOM-based XSS vulnerabilities.

Lab 3 — Tic-Tac-Toe Game 3 Hours

🎯 Lab 3 Objective

Build a complete two-player Tic-Tac-Toe game in the browser. The game must: render a 3×3 grid; alternate turns between Player X and Player O; detect all eight winning combinations and announce the winner; detect a draw when all cells are filled with no winner; highlight the winning cells; and include a reset button that fully clears the board for a new game. All logic is written from scratch using arrays and the DOM techniques from Day 1 theory.

Part A — Board Rendering & State (40 min)

The board is represented in JavaScript as an array of nine values — one for each cell. This array is the single source of truth. The DOM is always built from this array, never the other way around.

// tictactoe.js // The board: 9 cells, indexed 0-8, empty string means unclaimed let board = ["","","", "","","", "","",""]; let currentPlayer = "X"; let gameOver = false; // All eight ways to win: three in a row (row, column, or diagonal) const WIN_CONDITIONS = [ [0,1,2], [3,4,5], [6,7,8], // rows [0,3,6], [1,4,7], [2,5,8], // columns [0,4,8], [2,4,6] // diagonals ]; // Build the 9 cell divs from HTML — each gets an index as data attribute function renderBoard() { const grid = document.getElementById("board"); grid.innerHTML = ""; // safe here — no user input, we're building our own HTML board.forEach((value, index) => { const cell = document.createElement("div"); cell.classList.add("cell"); cell.textContent = value; // textContent — safe, no user input here cell.dataset.index = index; cell.addEventListener("click", handleCellClick); grid.appendChild(cell); }); } renderBoard();
Part B — Turn Logic & Win Detection (60 min)

Write the click handler, the win-checking function, and the draw-detection logic. Students should write checkWinner() independently first, then compare with the instructor's version.

// --- HANDLE A CELL CLICK --- function handleCellClick(event) { const index = parseInt(event.target.dataset.index); if (gameOver || board[index] !== "") return; // ignore invalid clicks board[index] = currentPlayer; // claim the cell in the array renderBoard(); // rebuild the DOM from the array const winningLine = checkWinner(); if (winningLine) { highlightWin(winningLine); setStatus("🎉 Player " + currentPlayer + " wins!"); gameOver = true; } else if (board.every(cell => cell !== "")) { setStatus("It's a draw!"); gameOver = true; } else { currentPlayer = (currentPlayer === "X") ? "O" : "X"; setStatus("Player " + currentPlayer + "'s turn"); } } // --- CHECK FOR A WINNER --- // Returns the winning [a, b, c] index triplet, or null if no winner yet function checkWinner() { for (const [a, b, c] of WIN_CONDITIONS) { if (board[a] !== "" && board[a] === board[b] && board[b] === board[c]) { return [a, b, c]; // return the winning line so we can highlight it } } return null; } // --- HIGHLIGHT WINNING CELLS --- function highlightWin(indices) { const cells = document.querySelectorAll(".cell"); indices.forEach(i => cells[i].classList.add("winning-cell")); } // --- STATUS MESSAGE --- function setStatus(message) { document.getElementById("status").textContent = message; } // --- RESET BUTTON --- document.getElementById("reset-btn").addEventListener("click", () => { board = ["","","", "","","", "","",""]; currentPlayer = "X"; gameOver = false; renderBoard(); setStatus("Player X's turn"); });

Lab 4 — Secure JavaScript Patterns & XSS Prevention 4 Hours

🎯 Lab 4 Objective

Students audit two deliberately vulnerable web applications, identify the XSS vulnerabilities in each, exploit them safely in a controlled environment, then rewrite the affected code using secure patterns. By the end, each student produces a working mini security report documenting what they found, how they exploited it, and what they fixed.

Part A — Live Exploit: innerHTML XSS (45 min)

The instructor provides a simple vulnerable search results page. Students type a crafted input into the search field and observe their own JavaScript execute on the page.

// ── The vulnerable code (instructor provides this file) ── const query = document.getElementById("search").value; document.getElementById("results").innerHTML = "Showing results for: " + query; // VULNERABLE // ── Students type each of these into the search box ── // Test 1 — basic alert payload <script>alert("XSS works")</script> // Result: alert appears — code executed // Test 2 — image onerror payload (works even when <script> is blocked) <img src=x onerror="alert('Still XSS')"> // Result: alert appears — event handler executed // Test 3 — cookie theft simulation (no real data, just logs to console) <img src=x onerror="console.log('Cookie: ' + document.cookie)"> // Result: any cookies on the page are logged — in a real attack, sent to attacker's server

Part D — Content Security Policy (CSP) — First Exposure (30 min)

Students add a Content Security Policy meta tag to both the calculator and form pages. A CSP tells the browser which sources of scripts, styles, and images are trusted — acting as a browser-level firewall against XSS. Even if an attacker injects a script, CSP can prevent it from executing.

<!-- Add this inside the <head> of both lab pages --> <meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"> <!-- What each directive means: default-src 'self' — only load resources from this same origin script-src 'self' — only run scripts from files on this origin inline <script> tags are BLOCKED by this policy style-src 'self' 'unsafe-inline' — allow inline styles (common during dev) --> // After adding the CSP, test whether the XSS payloads from Part A still execute // They should not — the browser refuses to run inline scripts // The DevTools console will show: "Refused to execute inline script"

Report Structure (write in a file called SECURITY.md and commit it):
  • Date and scope — what was audited and when
  • Findings table — one row per issue: file name, line number, vulnerability type, severity (Low / Medium / High), and status (Fixed / Accepted)
  • Description of each finding — what the code did, what an attacker could have done with it
